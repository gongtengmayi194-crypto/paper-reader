# paper-reader Skill 双报告重设计方案

**日期**: 2026-03-11  
**状态**: 已批准，待实施

---

## 背景与动机

现有 paper-reader skill 存在以下缺陷：
1. 只输出单一 `report.md`，论文解读与代码分析混杂在一起
2. 报告内容过短、不够详细，图例仅有简单批注，缺乏逐步流程描述
3. 代码分析部分只覆盖 2-3 个核心模块，覆盖不全
4. 没有测验机制，读者无法检验自己是否真正理解了论文

---

## 设计决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 报告结构 | 方案 A（主报告 + code-modules/ 子目录 + 理解度测试） | 模块化，代码部分可无限扩展，便于按需查阅 |
| 论文解读详细度 | 方案 B（逐层张量追踪） | 追踪每层的张量形状、参数量、计算细节，务必详尽 |
| 代码模块划分 | 按功能模块（数据→模型→训练→推理） | 与论文章节结构一致，便于对照 |
| 图例讲解风格 | 流程式讲解 | 按数据流动顺序逐步描述，配合张量形状变化 |
| 测验题型 | 混合题型 | 概念理解 + 流程追踪 + 代码实现 + 对比分析 |

## Report 1：论文深度解读.md 规格

### 整体原则
- 不求短而精炼，但求读懂而详细
- 每个图例必须按数据流向逐步描述，不只是"这张图展示了…"
- 全程使用 annotation-guide.md 的语言规范（避免"显然"等）

### 章节详细度分级

| 章节类型 | 详细度 | 具体要求 |
|----------|--------|----------|
| Introduction / Related Work | 中等 | 2-4 段说明背景、痛点、前人工作对照 |
| **Method（核心章节）** | **极致详细（方案 B）** | 逐层张量追踪，每个模块的输入输出形状 `(B,C,H,W)`，参数量计算，关键操作说明 |
| Experiments | 中等 | 解读每张结果表格，说明消融实验结论，指明性能提升来源 |
| Conclusion | 简述 | 1-2 段总结贡献与局限 |

### Method 章节图例讲解规格（方案 B 详细度）

每张架构图必须包含：

1. **整体类比**：把整个架构比作日常生活中熟悉的系统
2. **逐模块流程描述**（重点）：
   - 按数据流动顺序编号
   - 每步说明：操作名称 + 具体参数（卷积核大小、步长、填充）+ 输入输出张量形状
   - 示例格式：
     ```
     ① RGB 图像 (B, 3, 224, 224) 进入 Conv1
        → 3×3 卷积，64 个滤波器，stride=1，padding=1
        → 输出 (B, 64, 224, 224)
        → 参数量：3×64×3×3 = 1,728
     ② 经过 BatchNorm
        → 对每个通道独立归一化（均值0，方差1）
        → 可学习参数：γ 和 β 各 64 个
     ③ ReLU 激活
        → 负值置零，不引入新参数
     ④ 2×2 平均池化，stride=2
        → 输出 (B, 64, 112, 112)
        → 空间分辨率减半
     ```
3. **关键设计决策说明**：
   - 为什么用 concat 而非 add？
   - 为什么这里用平均池化而非最大池化？
   - 为什么需要 1×1 卷积降维？
4. **与其他模块的关系**：这个模块的输出去哪里

## Report 2：code-modules/ 规格

### 每个子文件的结构

```markdown
# {模块名} (对应论文 {章节号} 节)

## 模块概述
- 功能说明（1-2句）
- 在整体架构中的位置
- 输入/输出规格

## 代码位置
- 主要文件：`{文件路径}`
- 入口函数/类：`{类名或函数名}`

## 逐函数解读

### {函数名/类名}

**与论文的对应**：{论文章节和公式编号}

```{language}
# 完整代码，含行号
{code with line numbers}
```

**逐行注释**：
- **第 X-Y 行**：{详细说明，包含张量形状变化}
- **第 Z 行**：{关键操作说明}

**张量形状追踪**：
```
输入:  {shape}
经过 {操作}: {shape}
输出:  {shape}
```

**关键设计细节**：
- {易错点 1}
- {易错点 2}

**与论文的差异**（如有）：
- {论文描述 vs 代码实际实现的差异}
```

### 00-仓库总览与映射.md 特殊规格

必须包含：
1. 完整目录树（带注释）
2. 论文章节↔代码文件完整映射表
3. 各模块文件的快速跳转导航

### 模块数量与划分规则

- 按功能模块划分，不是按代码文件划分
- 最少 3 个模块（数据、模型、训练），最多按实际项目结构扩展
- 每个大型模块可进一步细分子节

## 理解度测试.md 规格

### 题型分布

| 题型 | 比例 | 示例 |
|------|------|------|
| 概念理解题 | 40% | "为什么使用 concat 而非 add 来融合双分支特征？" |
| 流程追踪题 | 30% | "描述一张 RGB 图像从输入到第一个融合层的完整流程，包括每步的张量形状" |
| 代码定位题 | 20% | "找出代码中计算多模态对比损失的具体函数和行号" |
| 对比分析题 | 10% | "与论文中引用的 Baseline X 相比，本文方法在哪个设计上有本质不同？" |

### 答案格式

每道题必须包含：
- 题目
- 参考答案（详细）
- 答案出处（论文章节 + 代码行号）
- 评分标准（关键知识点清单）

### 题目数量

- quick 模式：5 题
- core 模式：10 题  
- full 模式：15-20 题

## 对现有 SKILL.md 的修改点

### Phase 5（Generate Report）需改动

**原有**：
- 只生成 `report.md`
- 读取单一模板

**改为**：
1. 生成 `论文深度解读.md`（读取新模板 `report-template-paper-deep.md`）
2. 生成 `code-modules/` 下的各子文件（读取新模板 `report-template-code-module.md`）
3. 生成 `理解度测试.md`（读取新模板 `report-template-quiz.md`）

### 新增模板文件

| 文件 | 用途 |
|------|------|
| `references/report-template-paper-deep.md` | 论文深度解读主报告模板 |
| `references/report-template-code-module.md` | 单个代码模块报告模板 |
| `references/report-template-quiz.md` | 理解度测试模板 |

### 旧模板处理

旧模板文件已移除；`quick/core/full` 仅保留为深度档位参数（控制解读深度与题量）。

### parse_repo.py 需扩展

当前 `parse_repo.py` 输出：`repo-report.md`、`structure-tree.txt`、`key-functions.md`

需要扩展为按功能模块输出分析结果，供生成 code-modules/ 子文件使用：
- 输出 `code-analysis/module-{name}.json`，包含该模块的函数列表、代码片段、与论文的映射关系
